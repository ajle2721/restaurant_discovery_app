# Restaurant Discovery App Workspace Rule

## Purpose
Use this rule for all work in this repository. This is a static web app for discovering kid-friendly restaurants in Taipei. The live frontend is built from raw Google Places data in `response/` plus per-restaurant analysis files in `ai_review/`.

## Environment Assumption
- This rule is for the Windows development environment used with Antigravity, not the current Codex shell environment.
- Prefer Windows-friendly commands, paths, and tooling expectations.
- Do not assume `nix develop`, `just`, local HTTP servers, Unix shells, or Linux-specific utilities are available when following this rule inside Antigravity.

## Source Of Truth
- Treat `response/*.json` as the source of place metadata and review text.
- Treat `ai_review/*.json` as the source of restaurant-level analysis.
- Treat `ai_review/index.js` as a generated artifact, not a hand-edited source file.
- Do not reintroduce `data.js` or rebuild workflows around it.

## Active Build Flow
- Local dev flow on Windows:
  - Team members commonly preview by opening `index.html` directly from File Explorer.
  - When a generated frontend bundle must be refreshed, update the source data and ensure `ai_review/index.js` is regenerated before pushing.
  - If local execution tooling is unavailable, prefer preparing the correct file changes for commit and validating through GitHub after push.
- The frontend bundle is generated by `build-ai-review-index.mjs`.
- GitHub Pages deploy uses `.github/workflows/deploy.yml` and must continue to build `ai_review/index.js` before publishing.

## Editing Rules
- Prefer small, surgical edits that preserve the existing vanilla HTML/CSS/JS structure.
- Keep UI copy in Traditional Chinese unless the task explicitly requires otherwise.
- Use 4-space indentation in JS, Python, and HTML.
- Do not add frameworks, bundlers, or package-manager-based frontend tooling unless explicitly requested.
- If changing data logic, update the generator or the underlying JSON sources, not the generated bundle by hand.

## Validation
- After frontend changes, prefer a direct browser check by opening `index.html`.
- After data or build-pipeline changes, make sure the generated bundle `ai_review/index.js` is updated before pushing.
- If a change affects the generated bundle, mention that `ai_review/index.js` was rebuilt.

## Cleanup Boundaries
- It is safe to remove obsolete scripts tied to the old `data.js` workflow.
- Do not delete `response/` or `ai_review/` source data unless explicitly requested.
- Treat `.env.txt` and API keys as sensitive and never commit secrets.
